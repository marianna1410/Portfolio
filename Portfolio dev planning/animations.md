# Animations Specification — Marianna Portfolio

Цей файл описує **рух і візуальні переходи** на сайті: тривалість, easing, тригери, поведінка при hover, скролі, появі. Поведінка (що куди веде, що клікабельне) — у `interactions.md`. Доступність (включно з `prefers-reduced-motion`) — у `accessibility.md`.

Сайт — Astro. Анімаційний стек:
- **Прості (hover, transitions між станами, дрібні keyframes)** → CSS.
- **Складніші (orchestrated, scroll-tied, orbital motion, stack-swap)** → GSAP + ScrollTrigger + ScrollSmoother.
- **Page transitions** → Astro View Transitions API (вбудовано).

GSAP з 2024 повністю безкоштовний (включно з ScrollSmoother, ScrollTrigger та іншими плагінами), ліцензія не потрібна.

## 0. Загальні принципи

### 0.1. Reduced motion (обов'язково)

Усі анімації мають поважати `prefers-reduced-motion: reduce`. Конкретні правила — у `accessibility.md`. Коротко:
- transitions і decorative animations → `transition: none` / `animation: none`
- ScrollSmoother не ініціалізується — нативний браузерний скрол
- orbital motion (Hero icons) → іконки залишаються статичними у своїх позиціях
- page transitions → миттєвий перехід без fade
- dotted arrow у footer → крапки в стані opacity 1.0, без руху
- Travelling cards → без fan-out reveal-анімації при появі; клік-свап лишається

### 0.2. Performance-правила

Анімувати лише GPU-friendly властивості:
- `transform` (translate, scale, rotate)
- `opacity`
- `filter` (з обережністю)

Не анімувати: `width`, `height`, `top`, `left`, `margin`, `padding` — це викликає reflow і лагає на слабших пристроях.

### 0.3. Стандартні easing-функції

| Назва         | CSS-значення                        | Де використовується                            |
|---------------|-------------------------------------|------------------------------------------------|
| `ease-out`    | `cubic-bezier(0.0, 0.0, 0.2, 1)`    | Відкриття, hover-in, fan-out reveal            |
| `ease-in`     | `cubic-bezier(0.4, 0.0, 1, 1)`      | Закриття, hover-out                            |
| `ease-in-out` | `cubic-bezier(0.4, 0.0, 0.2, 1)`    | Симетричні рухи (orbit, stack-swap)            |
| `linear`      | `linear`                            | Loop-анімації (orbit) — щоб не було ривка на стику |

У CSS винести як змінні в `tokens.css`:
```css
--ease-out: cubic-bezier(0.0, 0.0, 0.2, 1);
--ease-in: cubic-bezier(0.4, 0.0, 1, 1);
--ease-in-out: cubic-bezier(0.4, 0.0, 0.2, 1);
```

### 0.4. Стандартні тривалості

| Тривалість | Де використовується                          |
|------------|----------------------------------------------|
| `100ms`    | Pressed/active feedback                      |
| `150ms`    | Hover-стани, дрібні переходи кольору/обводки |
| `200ms`    | Закриття дропдауну, transitions між станами  |
| `250ms`    | Hover-in, image cross-fade                   |
| `400ms`    | Page transitions                             |
| `500ms`    | Stack-swap (Travelling)                      |
| `600–800ms`| Fan-out reveal (Travelling)                  |
| `15–20s`   | Loop-анімації (orbit)                        |

Винести як змінні в `tokens.css`:
```css
--duration-instant: 100ms;
--duration-fast: 150ms;
--duration-medium: 250ms;
--duration-slow: 400ms;
```

## 1. Глобальні анімації

### 1.1. Smooth scroll — GSAP ScrollSmoother

> **❌ СКАСОВАНО (рішення Marianna, Phase 3).** ScrollSmoother НЕ використовуємо:
> він огортає контент у transform-обгортку, що ламає `position: sticky`
> (хедер-пілюля, TOC-сайдбар) і `position: fixed` (TOC FAB). Лишаємо
> **нативний smooth-scroll**: `scroll-behavior: smooth` + `scroll-margin-top`
> на якорях (вже працює, у `global.css`). Без momentum-ефекту — і без ризику
> для верстки. Текст нижче залишено для історії, не імплементувати.

**Реалізація:** GSAP ScrollSmoother (https://gsap.com/docs/v3/Plugins/ScrollSmoother/).

**Параметри:**
- `smooth: 1.2` — час "догону" скролу (секунди). Підкрутити в процесі (0.8–1.5).
- `effects: true` — підтримка `data-speed` і `data-lag` атрибутів на елементах для майбутніх ефектів.
- `normalizeScroll: true` — узгоджена поведінка між браузерами і мобілками.

**Інтеграція з кліками меню і TOC:**
- Клік з пункту меню (Works/Experience/About) → `scrollSmoother.scrollTo(target, true, "top 40px")` — offset 40px зверху.
- Клік з пункту TOC → `scrollSmoother.scrollTo(target, true, "top 20px")` — offset 20px зверху.

**Мобільна поведінка:**
ScrollSmoother рекомендують **вимикати на тач-девайсах** — нативний momentum-scroll iOS і Android краще, ніж штучний smoothing. Ініціалізувати тільки коли:
```js
window.matchMedia('(hover: hover) and (pointer: fine)').matches
```
На решті — звичайний браузерний скрол.

**Reduced motion:**
При `prefers-reduced-motion: reduce` — ScrollSmoother не ініціалізується взагалі, скрол стандартний браузерний (миттєвий до якоря).

### 1.2. Page transitions (між сторінками)

Astro View Transitions API (`<ViewTransitions />` у Layout).

**Ефект:**
- Стара сторінка: fade out (200ms `ease-in`)
- Нова сторінка: fade in (200ms `ease-out`)
- Сумарна тривалість: ~400ms
- Без `scale` чи `translate` — мінімалістично, не відволікає від контенту

Стосується переходів:
- головна ↔ кейс Lucida
- головна ↔ кейс Vaia
- Lucida ↔ Vaia (через Next case card)

## 3. Dropdown (кнопка Contact)

Параметри з Figma-анотації `Annotations - Header/home page/dropdown opened`.

### 3.1. Відкриття

- Початковий стан: `opacity: 0; transform: translateY(-8px);`
- Кінцевий стан: `opacity: 1; transform: translateY(0);`
- Тривалість: `200ms`
- Easing: `ease-out`

### 3.2. Закриття

- Початковий стан: `opacity: 1; transform: translateY(0);`
- Кінцевий стан: `opacity: 0; transform: translateY(-8px);`
- Тривалість: `150ms`
- Easing: `ease-in`

### 3.3. Іконка стрілки на кнопці-тригері

Синхронно з відкриттям/закриттям дропдауну.

- Закрита → Відкрита: `rotate(0deg) → rotate(180deg)`, `200ms ease-out`
- Відкрита → Закрита: `rotate(180deg) → rotate(0deg)`, `150ms ease-in`

Реалізація — CSS `transform: rotate()` на іконці, прив'язаний до атрибута `aria-expanded` кнопки.

## 4. Hero section

### 4.1. Orbital motion іконок (idle-стан)

**Концепція:** усі іконки (Notion, Claude, Figma, Google Calendar, Slack, Flash, Eye, Clever, Flower, Gemini) плавно рухаються по овалу — повторюючи еліптичну траєкторію, на якій вони розташовані у дизайні.

**Референс:** https://codepen.io/GreenSock/pen/NWeLyrq — адаптувати до **2D** (наш дизайн плоский, не 3D).

**Технічні параметри:**
- **Бібліотека:** GSAP.
- **Траєкторія:** еліптична, повторює стартові позиції іконок з Figma.
- **Напрямок:** **clockwise (за годинниковою стрілкою).**
- **Швидкість:** один повний оберт ~ `15–20s`. Підкрутити в процесі — має відчуватись повільно і медитативно.
- **Фазові зсуви:** кожна іконка стартує зі своєї позиції (того кута еліпса, на якому стоїть у Figma) — не з однієї точки.
- **Варіативність швидкості:** легке варіювання між іконками (±10–15% від базової), щоб не виглядало як одна обертова конструкція.
- **Loop:** infinite, `linear` easing (НЕ `ease-in-out` — буде ривок на стику циклу).
- **Старт:** orbital motion починається після завантаження сторінки (одразу, без затримки на reveal).

### 4.2. Hover на іконку

**При наведенні:**
1. **Уся orbital motion зупиняється** — усі іконки замирають на своїх поточних позиціях (`timeline.pause()`).
2. **Конкретна іконка під курсором плавно збільшується:**
   - `scale(1) → scale(1.15)` (підібрати в процесі: 1.1–1.2)
   - тривалість: `250ms`
   - easing: `ease-out`

**При виході курсора (hover end):**
1. Іконка повертається до `scale(1)`, `250ms ease-out`.
2. Orbital motion відновлюється **з поточних позицій** (`timeline.resume()`, НЕ `restart()`).

## 7. Behind the Screen section

### 7.1. Travelling — стек фото з fan-out reveal і click-swap

**Що бачимо в дизайні:** стопка з кількох фото, накладених одна на одну з невеликими поворотами.

**Анімація появи (fan-out reveal):**
- Тригер: коли секція з'являється у viewport (Intersection Observer або GSAP ScrollTrigger).
- Кожна картка стартує з центральної позиції стопки → виїжджає у свою фінальну позицію з власним поворотом.
- Stagger між картками: ~120ms (трохи помітно, щоб бачити "розкладання").
- Тривалість руху однієї картки: ~600ms, загалом fan-out ≈ 800ms.
- Easing: `ease-out`.
- Спрацьовує **один раз** на сесію (не повторюється при повторному скролі).

**Click-swap (інтерактив): ❌ СКАСОВАНО (рішення Marianna, Phase 3).**
Click-swap фото НЕ робимо. Лишається тільки fan-out reveal при появі (вище). Текст нижче залишено для історії, але не імплементувати.

**Референс:** https://penghsuanlee.com/about — секція з фото внизу сторінки, де клік перемикає верхню картку в стопці.

**Поведінка:**
- Клік на верхню картку → вона анімовано "відходить" і опускається в кінець z-стеку.
- Друга картка стає верхньою; інші зсуваються на одну позицію вгору в z-стеку.
- Стартові поворот/offset кожної картки зберігаються відносно її позиції в стеку (не картки як такої).

**Анімація переходу:**
- Верхня картка: `scale 1 → 0.95`, `opacity 1 → 0.7`, `translateY 0 → +20px` — за 250ms.
- Потім вона ж: змінює z-index на найнижчий, повертається `translateY → 0`, `scale → 0.92` (як у нижньої позиції стеку), `opacity → 1` — ще 250ms.
- Решта карток синхронно піднімаються на свою нову позицію в стеку.
- Загальна тривалість: `500ms ease-in-out`.
- Реалізація через GSAP timeline.

**Мобілка:**
- Тап на верхню картку = клік (та сама поведінка циклічного перемикання стека).
- Swipe-жести не реалізуємо (тап достатньо, не ускладнюємо).

## 9. Кнопки

Анімація hover — комбінація **анімації крапок у Figma-стрілці** і **переходу між Figma-станами** (default/hover/focus/disabled).

### 9.1. Стани кнопок з Figma (всі варіанти)

У дизайні **4 стани кожної кнопки** (для filled і outline варіантів):
- **Default** — базовий вигляд
- **Hover** — змінюється колір фону / тексту / обводки
- **Focus** — обводка focus-кольору (для клавіатурної навігації)
- **Disabled** — приглушений сірий

Переходи між станами:
- default ↔ hover: `150ms ease-out` (колір, фон, обводка)
- default → focus: статичний стан (без анімації, тільки CSS-перехід обводки за `100ms` — для плавності, але без декорування)
- → disabled: без анімації (стан змінюється миттєво при зміні стану кнопки в коді)

### 9.2. Анімація стрілки-з-крапок при hover

**Важливо:** стрілки у дизайні **вже існують у Figma** і мають власний візерунок з крапок. Ми **НЕ замінюємо** їх на стрілку з рефу 21st.dev — використовуємо ту, що в дизайні. Анімаційний **патерн** (sequential brightening) береться з рефу як концепція, **візуал** — з Figma.

**Стрілки можуть бути в різних напрямках:**
- → (forward, right)
- ← (back, left)
- ↗ (diagonal up-right)
- ↘ (diagonal down-right)
- ↖, ↙ (теж можливо, якщо з'являться)

Напрямок задається атрибутом, наприклад `data-arrow-direction="up-right"` на елементі стрілки. Анімаційний JS-скрипт визначає, у якому порядку анімувати крапки відповідно до напрямку.

**Реалізація:**
- Стрілка інлайн-SVG. Кожна крапка стрілки — окремий `<circle>` із власним `id` або порядковим номером.
- **Default state:** усі крапки з opacity ~0.6 (як у Figma default).
- **Hover (триває поки курсор на кнопці):**
  - крапки по черзі піднімають opacity з 0.6 до 1.0 і повертаються до 0.6
  - **порядок** залежить від напрямку стрілки:
    - → крапки brightенять зліва направо
    - ← справа наліво
    - ↗ з нижнього лівого кута до верхнього правого
    - ↘ з верхнього лівого до нижнього правого
  - sequential delay між сусідніми крапками: ~50ms
  - тривалість brightening однієї крапки (peak та повернення): ~200ms
  - анімація loops, поки hover активний
- **Hover-out:**
  - якщо анімація в середині циклу — дочекатись поточної крапки до повернення в 0.6, далі зупинка
  - усі крапки в default state протягом 200ms ease-out

**Active (pressed):**
- `transform: scale(0.98)` на всю кнопку
- тривалість: `100ms ease-out`
- після відпускання — повернення в hover-стан (якщо курсор лишається) або default

### 9.3. Disabled state

- Активується через атрибут `disabled` або клас `.is-disabled`.
- Візуал — з Figma disabled-стану.
- Жодних hover/active анімацій (pointer-events: none).
- Курсор: `cursor: not-allowed`.

### 9.4. Focus-visible (клавіатура)

- Стан focus з Figma-компонента (зазвичай це додаткова обводка).
- Тільки для клавіатурної навігації — `:focus-visible`, не `:focus` (щоб не показувати при кліку мишею).
- Без декоративної анімації.
- Деталі ARIA — у `accessibility.md`.

### 9.5. Кнопка Contact (з дропдауном)

Окрім стандарту 9.1–9.4:
- При відкритому дропдауні (`aria-expanded="true"`) — кнопка візуально в active-стані (інверсія або інший стан із Figma).
- Іконка стрілки обертається на 180° (див. 3.3).

## 10. Текстові посилання

### 10.1. Default state

Текст у дизайн-кольорі, **без підкреслення** на дефолті.

### 10.2. Hover — посилання БЕЗ стрілки

**Стосується:** пунктів навігації меню (Works / Experience / About), посилання Go Home, текстових частин у дропдауні.

- Зміна кольору тексту відповідно до Figma hover-стану.
- Тривалість: `150ms ease-out`.

### 10.3. Hover — посилання ЗІ стрілкою (зовнішні і внутрішні зі стрілкою)

**Стосується:** усіх посилань, у яких є стрілка з крапок у дизайні — footer-посилання (Email, LinkedIn, Telegram, Resume), у дропдауні Contact, інші Link-компоненти зі стрілкою.

**Той самий підхід, що для кнопок (9.2):** використовуємо стрілки з Figma (їхній dot-патерн), застосовуємо sequential brightening на крапках. Напрямок стрілки задається `data-arrow-direction` (як у 9.2) — посилання можуть мати стрілки в різних напрямках, не лише ↗.

**Параметри анімації стрілки** — ті самі, що в 9.2 (50ms delay, 200ms на крапку, opacity 0.6 → 1.0 → 0.6, loops поки hover).

**ДОДАТКОВО — Figma hover-стан текстової частини:**
- Колір тексту і стрілки змінюється відповідно до hover-стану з Figma.
- Тривалість зміни кольору: `150ms ease-out`.

### 10.4. Стани посилань (default / hover / focus / disabled)

Стани з Figma (бачимо в компонентах link):
- **Default** — звичайний колір
- **Hover** — зміна кольору і брайтенінг крапок стрілки (10.3)
- **Focus** — обводка focus-кольору навколо посилання (з Figma)
- **Disabled** — приглушений сірий, без інтерактиву

Переходи — як у кнопок (9.1).

### 10.5. Focus-visible

З Figma-компонента. Тільки `:focus-visible` (не `:focus`). Без декоративної анімації. Деталі ARIA — у `accessibility.md`.

## 11. Footer

### 11.1. Анімація лінії з крапок + пташки ("Ready to start building? ⋯⋯⋯⋯ 🪶 Let's get in touch")

**Концепція:** лінія з крапок **статична**, а пташка-наконечник наприкінці лінії — "жива": робить легке стрепенення кожні кілька секунд (як справжня пташка на дроті — сидить смирно, потім ворухнулась). Ця тиша лінії і ритм пташки робить блок медитативним, без миготіння.

**Лінія з крапок:**
- Усі крапки на однаковій opacity (відповідно до дизайн-токена — можливо 0.6 або інший).
- **Жодного руху**, жодного wave-ефекту.

**Пташка-наконечник:**

Анімація 3-в-1, виконується послідовно як одна loop-петля з паузами:

**Цикл (повторюється кожні `~4–5s`):**

1. **Пауза**: пташка статична — `2.5–3s` (це більша частина циклу, щоб не відволікала).

2. **Стрепенення (`~800ms` загалом):**
   - крила (верхня частина пташки): легкий "wing flap" — `translateY(0 → -2px → 0)` із пов'язаним легким `scale(1 → 1.05 → 1)` усієї пташки
   - корпус: дрібний `rotate(0 → -2deg → +2deg → 0)`
   - easing: `ease-in-out` (м'яка хвиля, не різко)
   - усе разом сумарно ~800ms

3. **Знову пауза до наступного циклу.**

**Реалізація:**
- Пташка — окремий SVG-group, виокремлений від лінії крапок (transform-origin: center пташки).
- GSAP timeline з `repeat: -1` і `repeatDelay: 3`.
- Альтернатива: CSS `@keyframes` з `animation-duration: 4s infinite`.

**Опційне покращення (якщо при перегляді відчуватиметься нудно):**
- Додати дуже субтільну "хвилю" на лінії (Варіант 2 з варіантів): крапки по черзі піднімають opacity з 0.6 до 0.8 (НЕ до 1.0 — щоб залишалось акцентом саме на пташці). Sequential delay 120ms, тривалість крапки 500ms. Хвиля синхронізована з циклом пташки — стартує разом зі стрепенінням.
- Це опційне — спочатку імплементуємо чистий Варіант 3, тоді дивимось у живому прев'ю, чи треба додати.

**Performance:**
- Анімація запускається тільки коли блок у viewport (ScrollTrigger або Intersection Observer). Поза viewport — pause.

**Reduced motion:**
- Лінія статична (як і завжди).
- Пташка статична — без стрепенення, без хвилі.

## 12. Що НЕ анімуємо (свідомі рішення)

Щоб Claude не пропонував зайвого:

- **Smooth-reveal на скролі (fade-in блоків при появі)** — НЕ робимо як глобальне правило. Зокрема НЕ треба для секцій Hero, Works, Experience, Behind the Screen, Footer (за винятком конкретного fan-out у Travelling, описаного в 7.1).
- **Картки не "піднімаються" на hover** (тільки image swap у Works/Next case).
- **Меню-пункти не мають кастомного hover-руху** крім стандарту 10.2.
- **Жодних кастомних курсорів.**
- **Жодного parallax-ефекту.**
- **Без preloader-анімацій** (контент показується одразу).
- **Без typewriter-ефектів** на заголовках.
- **Без cursor-trailing ефектів.**
